# E2EE и общие нюансы приложения

Краткий список реализованного и оставшихся ограничений.

---

## Реализовано

- **Выход / смена пользователя** — при выходе очищаются ключи E2EE и ключи чатов, кэш сообщений (Hive).
- **Удаление аккаунта** — очистка E2EE и кэша перед переходом на экран входа.
- **Смена пароля** — бэкап ключей перешифровывается новым паролем.
- **Расшифровка** — при любой ошибке показывается «[зашифровано]», сырой JSON не показывается.
- **Участник по инвайту / добавление в группу** — ключ отдаётся через `shareChatKeyWithNewMembers` при открытии чата или после добавления участников; новый участник может запросить ключ по WS (`requestChatKey` → рассылка `e2ee_request_key` → другие отдают ключ).
- **Поиск в E2EE-чатах** — сервер отдаёт последние сообщения с полным `content`; клиент расшифровывает и фильтрует по запросу, строит сниппет.
- **Кэш (Hive)** — в кэше хранится ciphertext; при чтении офлайн сообщения расшифровываются.
- **Уход / исключение из чата** — при `leaveChat` и `removeMemberFromChat` сервер удаляет запись в `chat_keys` для этого пользователя (с нового устройства ключ получить нельзя).
- **Ответ (reply)** — контент цитируемого сообщения расшифровывается в `_decryptOne` вместе с основным сообщением.
- **Медиа (фото)** — перед загрузкой изображение шифруется ключом чата (`uploadImage(..., chatId)`); при отображении (виджет `E2eeImage`, полноэкранный просмотр, галерея) ответ по URL при необходимости расшифровывается. Файлы (не изображения) пока без E2EE.
- **Редактирование (editMessage)** — новый текст шифруется перед отправкой (`editMessage(..., chatId)`); при получении по WS (`message_edited`) контент расшифровывается для отображения, в кэш пишется ciphertext.
- **Удаление сообщения / очистка чата** — после успешного `deleteMessage` и `clearChat` вызываются `LocalMessagesService.removeMessage` и `LocalMessagesService.clearChat`, чтобы кэш не содержал удалённых сообщений.

## Ограничения

### 1. Участник без E2EE-ключей

- Если у пользователя нет ключевой пары (старый аккаунт и т.п.), ключ чата ему не выдаётся — будет «[зашифровано]» до появления ключей (например, после входа с паролем и `ensureKeyPair`).

### 2. Web и secure storage

- На web `flutter_secure_storage` может быть менее надёжен, чем на iOS/Android — для жёстких требований к веб-версии учитывать отдельно.

### 3. Ротация ключа при уходе участника

- Ушедший участник не может получить ключ с сервера с нового устройства; локально ключ у него может остаться до переустановки/очистки. Полное «забыть» историю даёт только ротация ключа чата (новый ключ + перешифрование сообщений) — не реализовано.

### 4. Файлы (не изображения)

- Вложения по `file_url` не шифруются E2EE; E2EE только для текста и загружаемых изображений.

### 5. Пересылка (forward) в E2EE

- Сервер копирует сообщение (в т.ч. `content`) в целевые чаты. Контент зашифрован ключом **исходного** чата; в целевых чатах другой ключ — участники увидят «[зашифровано]». Полноценная пересылка в E2EE потребовала бы перешифрования контента ключом каждого целевого чата на клиенте (или отдельного потока на сервере).

### 6. Скачивание «оригинала» изображения

- При скачивании по `originalImageUrl` файл может быть в формате E2EE (JSON с ct/n). Сейчас загрузка по URL сохраняет байты как есть; при E2EE нужно расшифровывать перед сохранением на диск (аналогично E2eeImage при отображении).

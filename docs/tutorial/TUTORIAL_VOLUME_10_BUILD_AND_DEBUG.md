## 1. Команды Flutter: run и build

- **flutter pub get** — установить зависимости из pubspec.yaml.
- **flutter run** — запустить приложение на подключённом устройстве или эмуляторе (по умолчанию debug). Выбор устройства: `flutter run -d chrome`, `flutter run -d <device_id>` (flutter devices).
- **flutter build web** — собрать веб-версию (выход в build/web).
- **flutter build apk** — APK для Android (build/app/outputs/flutter-apk/).
- **flutter build ios** — для iOS (требуется Xcode и настройка подписи).
- **flutter clean** — очистить кэш сборки при странных ошибках.

---

## 2. Конфигурация и переменные окружения

- **Базовый URL API** в проекте задаётся через **--dart-define=API_BASE_URL=...** (см. ApiConfig). Пример: `flutter run --dart-define=API_BASE_URL=https://api.example.com`.
- В коде: `const String.fromEnvironment('API_BASE_URL', defaultValue: '...')`.
- Секреты (ключи API) не хранить в коде — использовать переменные окружения при сборке или серверную конфигурацию.

---

## 3. Режимы сборки: debug и release

- **Debug** — быстрая сборка, отладочная информация, assert'ы, удобно для разработки. По умолчанию у `flutter run`.
- **Release** — оптимизация, минимум отладочной информации. `flutter run --release` или `flutter build web/apk/ios` дают release.
- **kDebugMode** — константа из foundation; в release она false. Использовать для логов: `if (kDebugMode) print(...);`.

---

## 4. Подготовка к публикации (кратко)

- **Версия** в pubspec.yaml: version: "1.0.0+1" (версия + build number).
- **Android:** подпись (keystore), настройка build.gradle (minSdkVersion, targetSdkVersion). Официальная документация Flutter: «Deploy to Android».
- **iOS:** сертификаты и provisioning profile в Xcode, настройка в Xcode и в проекте Flutter. «Deploy to iOS».
- **Web:** после flutter build web развернуть содержимое build/web на любой хостинг (статичные файлы).

---

## 5. Логи и отладка в продакшене

- В release логи print не показываются пользователю; для сбора ошибок используют сервисы (Firebase Crashlytics, Sentry и т.д.) или свой лог на сервер.
- Глобальная обработка в main: **runZonedGuarded** и **FlutterError.onError** — в проекте уже настроены; в release можно отправлять стек в сервис ошибок.
- Не логировать токены и пароли; в kDebugMode выводить только безопасную отладочную информацию.

---

## 6. Частые проблемы при сборке

- **«Gradle build failed» / «CocoaPods»** — обновить зависимости, проверить версии в android/ и ios/, при необходимости flutter clean и заново flutter pub get.
- **«API_BASE_URL not found»** — при использовании fromEnvironment передать при сборке: --dart-define=API_BASE_URL=...
- **Белый экран на Web** — проверить консоль браузера (F12), базовый URL и CORS на сервере; проверить, что маршруты приложения корректны при загрузке с подпути.

## 7. Проверь себя

1. Собери проект для web и открой в браузере. Проверь, что запросы уходят на правильный API (вкладка Сеть в DevTools).
2. Запусти с другим API_BASE_URL через --dart-define и убедись, что приложение обращается к новому адресу.
3. Где в проекте обрабатываются необработанные исключения (main.dart)? Что происходит при ошибке в release?

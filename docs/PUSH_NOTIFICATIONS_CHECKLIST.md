# Пошаговая проверка push-уведомлений

Если уведомления не приходят — проходи пункты по порядку и отмечай, что уже сделано.

---

## Шаг 1: Устройство получателя (на кого ждёшь пуш)

- [ ] **Приложение установлено с реальной сборки** (не веб, не эмулятор без push). Лучше тестировать на физическом iPhone/Android.
- [ ] **Вход выполнен под тем аккаунтом, который должен получать уведомления** (получатель).
- [ ] **Разрешения на уведомления включены:**  
  Настройки → [ваше приложение] → Уведомления — разрешить. При первом запуске не нажимали «Запретить».
- [ ] После входа открыт главный экран (список чатов) **минимум 5–10 секунд** — за это время приложение отправляет FCM-токен на сервер.

---

## Шаг 2: Токен в базе данных

В БД выполни:

```sql
SELECT id, email, fcm_token IS NOT NULL AS has_token
FROM users
WHERE email = 'vlad';  -- подставь email получателя
```

- [ ] У пользователя-получателя **`has_token = true`**.  
  Если `false` — токен на сервер не пришёл. Тогда: зайди в приложение под этим пользователем на телефоне, разреши уведомления, подожди на главном экране 10 сек, перезапусти приложение и проверь запрос снова.

---

## Шаг 3: Сервер (Yandex Cloud / переменные окружения)

На машине, где крутится бэкенд, в переменных окружения заданы:

- [ ] **`FIREBASE_PROJECT_ID`** (например `chat-app-43a67`)
- [ ] **`FIREBASE_CLIENT_EMAIL`** (из JSON ключа Firebase)
- [ ] **`FIREBASE_PRIVATE_KEY`** (приватный ключ из JSON, в одну строку с `\n` вместо переносов, или многострочно в кавычках)

После изменения переменных сервер **перезапущен**.

Проверка: в логах сервера при старте не должно быть строки  
`Firebase Admin: missing env` или `Push skipped: Firebase not configured`.  
Если при первой отправке сообщения в логах есть `Push skipped: Firebase not configured` — переменные не подхватились или сервер не перезапускали.

---

## Шаг 4: Миграция БД

- [ ] Выполнена миграция **`migrations/add_fcm_token.sql`** (колонка `users.fcm_token` существует):

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS fcm_token TEXT;
```

Проверка: в таблице `users` есть колонка `fcm_token`.

---

## Шаг 5: Firebase и APNs (iOS)

- [ ] В проекте есть **`ios/Runner/GoogleService-Info.plist`** (скачан из Firebase, добавлен в Xcode).
- [ ] В **Firebase Console** для iOS-приложения загружен **APNs Authentication Key** (.p8), указаны Key ID и Team ID.
- [ ] В **Xcode** у таргета Runner включена capability **Push Notifications** (файл `Runner.entitlements` содержит `aps-environment`).

---

## Шаг 6: Тестовый сценарий

1. **Устройство A:** войди под пользователем **А** (отправитель). Токен у А не обязателен.
2. **Устройство B:** войди под пользователем **Б** (получатель). У **Б** в БД должен быть `fcm_token` (п. 2).
3. На **устройстве B** сверни приложение (или закрой, но не выходи из аккаунта).
4. С **устройства A** отправь сообщение в чат, где есть оба А и Б.
5. На **устройстве B** должно прийти push-уведомление.

Если не пришло — смотри логи сервера в момент отправки сообщения (п. 7).

---

## Шаг 7: Логи сервера при отправке сообщения

При отправке сообщения в логах бэкенда должно быть что-то из этого:

- **`Push: otherMemberIds= [...] tokens found= 1`** — у получателя найден токен, пуш отправляется.
- **`FCM push: { successCount: 1, failureCount: 0 }`** — FCM принял отправку (уведомление должно дойти до устройства).
- **`FCM push: { successCount: 0, failureCount: 1 }`** и **`FCM token failure: ...`** — ошибка доставки (неверный/устаревший токен, APNs отклонил и т.д.).
- **`Push: tokens found= 0`** — у получателя в БД нет токена (вернись к шагу 2).
- **`Push skipped: Firebase not configured`** — на сервере не заданы или не подхвачены переменные Firebase (шаг 3).

По этим строкам можно понять, на каком этапе проблема.

---

## Краткая схема

```
[Отправитель] пишет в чат
       →
[Сервер] сохраняет сообщение, смотрит участников чата (кроме отправителя),
         берёт их fcm_token из users, вызывает FCM
       →
[FCM] шлёт уведомление на APNs (iOS) / напрямую (Android)
       →
[Устройство получателя] показывает push
```

Токен нужен **у получателя** (в БД у того пользователя, кому должно прийти уведомление).

# Отчёт по проверке безопасности приложения (Security Audit)

**Дата:** 2026-03-18  
**Объект:** full-stack приложение (Flutter + Node.js/Express + PostgreSQL + Firebase + Yandex Object Storage)  
**Метод:** статический анализ кода и конфигурации по чек-листу из 10 блоков.

---

## 1. Разведка и картография

### 1.1 Точки входа

**REST API (Express):**
- `GET /healthz` — health check (публичный)
- `GET /version` — версия приложения и ссылки на сторы (публичный)
- `GET /_meta` — метаданные сервера (только при `NODE_ENV !== 'production'`)
- **Auth:** `POST /auth/register`, `POST /auth/login` (публичные); `GET /auth/me`, `PATCH /auth/me`, `POST /auth/me/avatar`, `GET /auth/users`, `GET /auth/users/:userId`, `DELETE /auth/user/:userId`, `PUT /auth/user/:userId/password`, `POST /auth/unlock-private`, `POST /auth/fcm-token` (с JWT)
- **Chats:** `GET/POST/DELETE /chats`, `GET/POST/PUT/DELETE /chats/:id`, папки, инвайты, участники, роли — все с JWT
- **Messages:** `GET/POST/PUT/DELETE /messages/*`, загрузка изображений/файлов, реакции, пины, прочтения — с JWT
- **Students:** CRUD по студентам, балансы, транзакции, уроки — с JWT + `requirePrivateAccess` или `requireSuperuser` где нужно
- **Reports:** отчёты и занятия — `authenticateToken` + `requirePrivateAccess`; суперпользователь — отдельные эндпоинты
- **Bank statement:** загрузка выписки и применение платежей — `authenticateToken` + `requireSuperuser`
- **Setup:** `POST /setup/cors` — `authenticateToken` + `requireSuperuser`
- **Admin:** экспорт учёта, сброс пароля пользователя — `authenticateToken` + `requireSuperuser`
- **Moderation:** жалоба на сообщение, блок пользователя, список заблокированных — с JWT

**WebSocket:** один endpoint (тот же HTTP-сервер); аутентификация через заголовок `Authorization: Bearer <token>` или query `token=`. Подписка на чаты, typing, presence.

**Клиент (Flutter):** один базовый URL API из `lib/config/api_config.dart` (по умолчанию `https://reollity.duckdns.org`), переопределяется через `--dart-define=API_BASE_URL=...` при сборке. Глубокие ссылки/интенты в коде не картографировались (можно дополнить отдельно).

### 1.2 Роли и уровни доступа

| Роль | Как определяется | Доступ |
|------|------------------|--------|
| Аноним | нет JWT | Только `/auth/login`, `/auth/register`, `/healthz`, `/version` |
| Пользователь с JWT | любой залогиненный | Чаты, сообщения, профиль, список пользователей, модерация (жалобы, блоки) |
| Private access | JWT + `privateAccess: true` или в списке `PRIVATE_ACCESS_USERNAMES`/`PRIVATE_ACCESS_USER_IDS` | Отчёты, студенты, уроки, балансы (в рамках привязки teacher_students) |
| Суперпользователь | `SUPERUSER_USERNAMES` / `SUPERUSER_USER_IDS` в env | Сброс пароля, экспорт учёта, загрузка выписки, применение платежей, список отчётов, снятие пометки «поздний отчёт», полный доступ к студентам/транзакциям |

Сервисные аккаунты: только Firebase (FCM) через ключ в env; прямой сервисный доступ к API не предусмотрен.

### 1.3 Потоки данных

- **Клиент → API:** HTTPS (ожидается на проде), JWT в `Authorization`, тело JSON, для части запросов — `Idempotency-Key`, `X-Client-Timezone`.
- **API → БД:** все запросы через `pg` с параметризованными запросами (`$1`, `$2`, …).
- **API → хранилище:** загрузка файлов/аватаров в Yandex Object Storage (ключи из env).
- **API → Firebase:** отправка push через `firebase-admin` (учётные данные из env или файла).
- **WebSocket:** тот же сервер; рассылка по подпискам на чаты, данные из БД.

### 1.4 Критичные данные

- **PII:** логин (email в БД), пароль (хеш), display_name, аватар, timezone, FCM-токен, содержимое чатов и сообщений, отчёты и занятия, студенты (имя, контакты).
- **Секреты:** JWT_SECRET, ключи БД, Yandex, Firebase (в env; в репозитории не хранятся — `.env` в `.gitignore`).
- **Бизнес:** балансы студентов, транзакции, привязка teacher_students; доступ к отчётам по `created_by`.

## 2. Аутентификация и управление сессией

### 2.1 Пароли

- **Хеширование:** bcrypt, 10 раундов (`authController.js`: `bcrypt.hash(password, 10)`). Пароли в БД только в виде хеша.
- **Миграция с открытого пароля:** при первом успешном входе со старым паролем он перехешируется и сохраняется в БД — ок.
- **Валидация:** длина 6–128 символов, логин 4–50, формат логина `[a-z0-9_-]+` (`utils/validation.js`).

### 2.2 JWT

- **Генерация:** `jwt.sign(..., JWT_SECRET, { expiresIn: '7d', algorithm: 'HS256' })` — явный алгоритм, срок 7 дней (`middleware/auth.js`).
- **Проверка:** `jwt.verify(token, JWT_SECRET, { algorithms: ['HS256'] })` — защита от alg:none.
- **В payload:** `userId`, `email` (логин), `username`, `privateAccess`. Роль суперпользователя не в токене — проверяется на сервере по env при каждом запросе.
- **Ограничение длины токена:** до 4096 символов — защита от DoS (`middleware/auth.js`).
- **Refresh / инвалидация:** отдельного refresh-токена нет; при смене пароля и удалении аккаунта старый JWT остаётся валидным до истечения срока (типичный риск: украденный токен живёт до 7 дней). Рекомендация: при смене пароля/удалении аккаунта выдавать новый токен и по возможности хранить blacklist или версию токена.

### 2.3 Защита от перебора

- **Логин:** rate limit по паре `(username, IP)` — 5 неудачных попыток за 15 минут, только неуспешные считаются (`authLimiter` в `index.js`).
- **Регистрация:** тот же `authLimiter` на `/auth/register`.
- **Unlock private:** до 10 попыток за 15 минут на пользователя (`unlockLimiter`).
- **Смена пароля / удаление аккаунта:** до 5 попыток в час на пользователя (`sensitiveActionLimiter`).
- Капча не используется — при необходимости можно добавить на логин/регистрацию.

### 2.4 Компрометация одного JWT

- Доступ в рамках этого пользователя: чаты, где он участник, сообщения, профиль, отчёты/студенты — если у него есть private access. Подмена роли в токене не даёт прав суперпользователя (проверка по env). Но `privateAccess` в токене доверяется — при утёке кода доступа (unlock-private) злоумышленник может получить токен с `privateAccess: true` и видеть отчёты/студентов.

### 2.5 Восстановление пароля

- Самостоятельного «забыл пароль» через ссылку в письме нет. Есть только сброс пароля админом: `POST /admin/reset-user-password` (суперпользователь), тело `{ username, newPassword }`. Валидация пароля есть; логин нормализуется. Риск: при утёке учётки суперпользователя можно менять пароли других пользователей.

### 2.6 OAuth / соцсети

- Не используются. Только логин/пароль и JWT.

## 3. Авторизация и контроль доступа

### 3.1 Проверка прав на сервере

- Чаты и сообщения: везде используется `ensureChatMember(chatId, userId)` — доступ только к чатам, где пользователь в `chat_users`. Проверка по `chat_id` и `user_id` из JWT.
- Отчёты: `getReport`/`updateReport`/`deleteReport` проверяют `created_by = userId` для не-суперпользователя; суперпользователь видит все отчёты.
- Студенты: для не-суперпользователя проверяется связь `teacher_students(teacher_id, student_id)`; суперпользователь имеет полный доступ к студентам и транзакциям.
- Удаление/смена пароля аккаунта: только для своего `userId` (сравнение `req.user.userId` с `req.params.userId`).
- Админские и бухгалтерские эндпоинты защищены `requireSuperuser`.

### 3.2 IDOR

- Сообщения, чаты, отчёты, студенты — доступ привязан к членству в чате, `created_by` или `teacher_students`. Подмена только ID без членства/связи возвращает 403/404. Явных IDOR по коду не видно.
- `GET /auth/users/:userId` — любой аутентифицированный пользователь может запрашивать профиль по ID; возвращаются только id, username, displayName, avatarUrl (без пароля и чувствительных полей). Это может использоваться для перебора существующих userId (user enumeration) — см. раздел 9.

### 3.3 Горизонтальная/вертикальная эскалация

- Горизонтальная: доступ к чужим чатам/сообщениям/отчётам/студентам закрыт проверками членства и владения.
- Вертикальная: роль суперпользователя задаётся только в env, не из клиента; в JWT хранится только `privateAccess`, суперпользователь определяется по `isSuperuser(req.user)` на каждом запросе. Массовое присвоение полей (mass assignment) в контроллерах не замечено — обновляются только ожидаемые поля (например, display_name, timezone, а не role или флаги прав).

### 3.4 Скрытые/debug эндпоинты

- `GET /_meta` отдаёт метаданные (в т.ч. наличие конфига БД, Firebase, CORS) только при `NODE_ENV !== 'production'`; в проде возвращается 404. Рекомендация: не разворачивать прод с `NODE_ENV !== 'production'`.

## 4. Защита данных (в покое и в движении)

### 4.1 Шифрование в БД

- Чувствительные поля (пароли, токены) не шифруются дополнительно на уровне приложения — пароли хранятся в виде bcrypt-хеша. Рекомендация: шифрование БД на уровне диска (TDE) или ОС — на усмотрение инфраструктуры.

### 4.2 TLS

- В коде принудительное перенаправление на HTTPS не включено (это обычно делает обратный прокси/балансировщик). В проде заголовок `Strict-Transport-Security` выставляется (`index.js`). Смешанный контент — не применимо к чистому API; клиент должен обращаться по HTTPS (базовый URL в конфиге — https).

### 4.3 Хранение на клиенте (Flutter)

- Токен: на mobile/desktop — `FlutterSecureStorage`; на web — `SharedPreferences` (в коде явно указан риск при XSS: токен может быть прочитан скриптом; для высоких требований — httpOnly cookie).
- Остальные данные пользователя (userId, email, displayName, avatarUrl, настройки) — в SharedPreferences. Пароль на клиенте не хранится.

### 4.4 Логи

- В audit только тип события, IP, userId, без паролей и токенов (`utils/auditLog.js`). В логах при сохранении FCM пишется только `userId` и длина токена, не сам токен. В коде нет явного логирования полных тел запросов с паролями; при ошибках в проде в ответ клиенту отдаётся общее «Ошибка сервера» без деталей.

## 5. Ввод и инъекции

### 5.1 SQL

- Запросы к БД выполняются через `pool.query(sql, [param1, param2, ...])` с плейсхолдерами `$1`, `$2`. Конкатенация пользовательского ввода в SQL не обнаружена. Динамические фрагменты (например, `whereBefore`) формируются из числовых/контролируемых значений, не из сырого ввода.

### 5.2 Санитизация и XSS

- Отображаемые строки (ник, имя чата, папки) проходят `sanitizeForDisplay` (удаление управляющих и zero-width символов, ограничение длины). Текст сообщений — `sanitizeMessageContent` (удаление опасных управляющих символов). API отдаёт JSON; если контент показывается в WebView/HTML без экранирования — риск XSS на стороне клиента. Рекомендация: на веб-клиенте при выводе использовать экранирование/санитизацию (например, шаблоны с авто-экранированием).

### 5.3 Загрузка файлов

- **Изображения:** белый список расширений и MIME; SVG исключён (риск XSS). Лимит 10 MB, до 2 файлов. Файл в памяти затем загружается в Yandex (`utils/uploadImage.js`).
- **Файлы в чате:** белый список расширений и MIME-префиксов; лимит 100 MB, 1 файл. Имя файла санитизируется (basename, без path traversal, ограничение длины) (`utils/uploadFile.js`).
- **Выписка банка:** CSV/Excel; в csv-parser блокируются ключи `__proto__`, `prototype`, `constructor` — защита от prototype pollution. Лимит 10 MB.

Проверки по magic bytes для загрузок в коде не просматривались — рекомендация: при необходимости проверять сигнатуры файлов для критичных типов.

### 5.4 Опасные функции

- В коде приложения (не в node_modules) не используются `eval`, `exec`, динамическая сборка SQL из строк. Используются параметризованные запросы и санитизация.

## 6. Бизнес-логика и целостность

### 6.1 Идемпотентность

- Используется механизм idempotency: заголовок `Idempotency-Key` или поле `idempotency_key` в теле; ключ привязывается к `user_id`, `scope` и хешу тела; при повторе возвращается сохранённый ответ (`utils/idempotency.js`). Применяется к созданию отчётов, занятий, студентов (в контроллерах вызываются `beginIdempotent`/`completeIdempotent`). Это снижает риск дублирования операций при повторной отправке.

### 6.2 Гонки и транзакции

- Критичные операции (удаление аккаунта, создание/обновление отчётов, занятия, студенты, применение платежей) выполняются в транзакциях PostgreSQL (`BEGIN`/`COMMIT`/`ROLLBACK`). Для отчётов есть проверка уникальности даты в рамках пользователя. Блокировки по необходимости используются через `SELECT ... FOR UPDATE` в idempotency.

### 6.3 Валидация границ

- Длины полей ограничиваются (сообщения, отчёты, логин, пароль, имя файла и т.д.). Числовые ID парсятся через `parsePositiveInt` с проверкой на переполнение. Отрицательные суммы в бизнес-логике не проверялись точечно — стоит убедиться, что суммы платежей/балансов не могут быть отрицательными там, где это недопустимо.

## 7. Инфраструктура и конфигурация

### 7.1 Сети и firewall

- Из кода не следует — типично БД не выставляется в интернет, доступ только с сервера приложения. Рекомендация: описать в инфраструктурной документации (DMZ, порты, доступ к БД).

### 7.2 Секреты

- Секреты берутся из `process.env` (dotenv); `.env` в `.gitignore`. В репозитории есть `.env.example`/документация без реальных ключей. Жёстко закодированных пародов/ключей в коде не видно. В проде при старте проверяется наличие и длина `JWT_SECRET` (не короче 32 символов).

### 7.3 Зависимости

- Используются актуальные версии (express, pg, jsonwebtoken, bcryptjs, helmet, express-rate-limit и др.). Рекомендация: периодически запускать `npm audit` и обновлять зависимости при появлении CVE.

### 7.4 Заголовки безопасности

- Helmet включён (CSP отключён, т.к. API не отдаёт HTML). Вручную выставляются `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`, `Permissions-Policy`, в production — `Strict-Transport-Security`. `X-Powered-By` отключён.

## 8. Мобильный клиент (Flutter)

### 8.1 Обфускация и извлечение строк

- Стандартная сборка Flutter без явной обфускации в коде; при публикации в сторы рекомендуется включать обфускацию (например, `flutter build apk --obfuscate --split-debug-info=...`).

### 8.2 Утечка ключей и URL

- Базовый URL API задаётся в `lib/config/api_config.dart`: константа по умолчанию `https://reollity.duckdns.org` и переопределение через `String.fromEnvironment('API_BASE_URL', ...)`. Секреты API (ключи Firebase и т.п.) на клиенте обычно нужны только для push/аналитики; в коде не искались — убедиться, что ключи, не предназначенные для клиента, не попадают в билд.

### 8.3 Локальное хранилище

- Токен на mobile/desktop в FlutterSecureStorage; на web — SharedPreferences (риск при XSS отмечен в коде). Очистка при выходе выполняется.

### 8.4 Certificate pinning

- В коде не обнаружено pinning сертификата или публичного ключа сервера. Трафик полагается на системные сертификаты. При высоких требованиях к защите от MITM (особенно в корпоративных сетях) можно добавить pinning.

## 9. Продвинутые сценарии

### 9.1 Целенаправленная компрометация пользователя

- Восстановление пароля по ссылке не реализовано — сценарий не применим. Перехват OAuth не применим. При XSS на веб-клиенте возможна кража токена из SharedPreferences (в коде это отмечено). Рекомендация: для веб-версии рассмотреть выдачу токена в httpOnly cookie и запросы с credentials.

### 9.2 Компрометация одного серверного компонента

- При выполнении кода на сервере приложения злоумышленник получает доступ к процессу Node.js и переменным окружения (в т.ч. JWT_SECRET, DATABASE_URL, ключи Yandex/Firebase). Этого достаточно для полного доступа к БД и подмены JWT. Важно: изоляция сети, ограничение прав процесса, секреты в менеджере (Vault и т.п.) уменьшают ущерб.

### 9.3 Цепочки уязвимостей

- User enumeration через `GET /auth/users/:userId`: разные ответы 404 при несуществующем ID и 200 с профилем при существующем позволяют перебирать ID. Само по себе низкий риск; в сочетании с слабыми паролями или утечкой хешей может усилить атаку. Рекомендация: для не-суперпользователей возвращать один и тот же формат ответа (например, 404 с нейтральным сообщением) или ограничить частоту запросов по userId (уже есть getUserByIdLimiter).

### 9.4 Злоупотребление бизнес-логикой

- Лимиты rate limit и проверки прав ограничивают массовые действия. Идемпотентность снижает дублирование финансовых операций. Рекомендация: явно проверять знак и верхние границы сумм при пополнении/списании и в отчётах.

### 9.5 Утечка через побочные каналы

- Логин при неверном пароле: единое сообщение «Неверный логин или пароль» — тайминг по ответу БД может слегка отличаться (существующий пользователь vs несуществующий); при необходимости — константное время сравнения (для пароля уже используется bcrypt.compare). Различие в ответах по userId описано выше (user enumeration).

| 5 | Эндпоинт /_meta в non-production раскрывает конфиг | Инфо | Ранее закрыто | Не запускать прод с NODE_ENV !== 'production' (уже реализовано) |
| 6 | Капча на логин/регистрацию отсутствует | Низкая | Открыто | При росте брутфорса/ботов — добавить капчу или аналог |

### 10.2 Детали устранения (2026-03-19)

#### Уязвимость #1: Инвалидация JWT при смене пароля

**Механизм:** `token_version` (целое число в таблице `users`, по умолчанию 0).

- В JWT добавлено поле `tv` (token version), которое сверяется с `users.token_version` на каждом запросе в `authenticateToken`.
- При смене пароля (`changePassword`) — `token_version` инкрементируется, клиенту возвращается новый JWT.
- При сбросе пароля админом (`adminResetUserPassword`) — `token_version` инкрементируется, все сессии пользователя инвалидируются.
- При удалении аккаунта (`deleteAccount`) — `token_version` инкрементируется перед удалением записи.
- WebSocket-соединения проверяют `token_version` при установке — подключение с устаревшим токеном отклоняется.

**Файлы:**
- `migrations/add_token_version.sql` — миграция БД
- `middleware/auth.js` — проверка `tv` при аутентификации, генерация с `tv`
- `controllers/authController.js` — инкремент `token_version` при смене/сбросе пароля
- `websocket/websocket.js` — проверка `tv` при подключении
- `lib/services/auth_service.dart` — сохранение нового токена после смены пароля

**Обратная совместимость:** токены без поля `tv` считаются версией 0; если в БД `token_version = 0` (пароль не менялся с новым кодом), старые токены принимаются.

#### Уязвимость #3: User enumeration

**Механизм:** `getUserById` теперь проверяет наличие хотя бы одного общего чата между запрашивающим и целевым пользователем.

- Если общего чата нет — возвращается `404 Пользователь не найден` (тот же ответ, что и для несуществующего ID).
- Запрос своего профиля по ID работает без ограничений.
- Rate limiter (`getUserByIdLimiter`: 120 запросов/15 мин) сохранён как дополнительная защита.

### 10.3 Оставшийся план (приоритет)

1. **Средний (если есть web):** Хранение токена для веб в httpOnly cookie или снижение риска XSS.
2. **Низкий:** При высоких требованиях к безопасности — certificate pinning в мобильном клиенте.
3. **Низкий:** При росте брутфорса/ботов — добавить капчу на логин/регистрацию.

### 10.4 Быстрые победы

- Регулярно запускать `npm audit` и обновлять зависимости.
- Убедиться, что БД и бэкенд доступны только через защищённую сеть и что секреты не попадают в репозиторий.
- **Применить миграцию:** `node scripts/run-migration.js migrations/add_token_version.sql`

### 10.5 Теоретические сценарии (высокий уровень атакующего)

- **Кража JWT с устройства:** при физическом доступе или вредоносном ПО токен можно извлечь из secure storage; ограничение срока жизни токена и смена пароля с инвалидацией снижают окно. **Улучшено:** после смены пароля украденный токен мгновенно перестаёт работать.
- **Компрометация сервера:** при RCE на сервере приложения доступ к env и БД полный; смягчение — изоляция, минимальные права, секрет-менеджер, аудит доступа.
- **Цепочка IDOR + утечка:** в текущей реализации IDOR по чатам/отчётам/студентам закрыт проверками; **перебор userId через профили закрыт** проверкой членства в общих чатах.

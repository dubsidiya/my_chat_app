# Отчёт по проверке безопасности приложения

**Дата проверки:** 10 марта 2025  
**Объект:** Backend (Node.js, Express, PostgreSQL), Flutter-клиент, инфраструктура.

---

## Резюме

Проведён аудит аутентификации и авторизации, инъекций (SQL/NoSQL), валидации и санитизации ввода, работы с секретами, маршрутов и прав доступа, сессий/токенов, заголовков безопасности и логирования. Критических уязвимостей, позволяющих удалённо скомпрометировать данные без учётных данных, не обнаружено. Ниже перечислены найденные замечания и рекомендации.

---

## Что сделано хорошо

- **JWT:** явно задан алгоритм `HS256`, ограничена длина токена (4096), в production проверяется наличие и длина `JWT_SECRET` (≥32 символа).
- **Пароли:** bcrypt, миграция старых паролей в хеш при первом успешном входе; код разблокировки приватного доступа сравнивается через `crypto.timingSafeEqual`.
- **SQL:** запросы к БД выполняются через параметризованные вызовы (`$1`, `$2`, …), конкатенация пользовательского ввода в текст запросов не используется.
- **Доступ к данным:** доступ к чатам/сообщениям проверяется через `ensureChatMember`; отчёты фильтруются по `created_by`; студенты/уроки привязаны к `teacher_id`/`created_by`; админские маршруты защищены `requireSuperuser`.
- **Ввод:** используется `parsePositiveInt` для ID в критичных местах (moderation, частично chats); санитизация сообщений (`sanitizeMessageContent`), отображаемых строк (`sanitizeForDisplay`); ограничения длины (сообщения, логин, пароль, FCM-токен).
- **Загрузка файлов:** тип по MIME и расширению, лимиты размера, SVG исключён из изображений (риск XSS); имена файлов санитизируются (path traversal, null-byte); в bank statement — защита от prototype pollution в заголовках CSV.
- **Инфраструктура:** Helmet, X-Frame-Options, Referrer-Policy, в production — HSTS; rate limiting (глобальный, auth, API, upload); лимит размера тела запроса; CORS с явным списком/паттернами origin; в production обязательны `ALLOWED_ORIGINS` и `JWT_SECRET`.
- **Секреты:** нет хардкода паролей/ключей в коде; скрипт `check-no-secrets-staged.sh` не даёт коммитить файлы с секретами; аудит безопасности логирует события в production в JSON без PII.
- **Flutter:** токен на mobile/desktop хранится в `flutter_secure_storage`; на web — в SharedPreferences с явным комментарием о риске XSS; API URL задаётся через `ApiConfig`/dart-define, жёстко не зашиты ключи API.

## Найденные замечания и рекомендации

### 1. ~~Валидация `userId` в GET /auth/users/:userId (низкий риск)~~ ✅ Исправлено

**Файл:** `controllers/authController.js`, `getUserById`.

**Суть:** `userId` берётся из `req.params.userId` и передаётся в запрос как есть. При нечисловом значении PostgreSQL может вернуть 0 строк или ошибку, что приводит к 404 или к раскрытию деталей ошибки.

**Рекомендация:** использовать `parsePositiveInt(req.params.userId)` и при `null` возвращать `400` с сообщением «Некорректный ID пользователя». **Реализовано.**

---

### 2. ~~Раскрытие кодов ошибок БД в ответах (низкий риск)~~ ✅ Исправлено

**Файлы:** `controllers/reportsController.js` (около строк 671, 932).

**Суть:** В тексте ответа при ошибке создания/обновления отчёта возвращается `error?.code` (например, код PostgreSQL). Это может помочь атакующему уточнять тип БД и характер ошибок.

**Рекомендация:** в production не включать `error?.code` в тело ответа; логировать код и стек только на сервере и возвращать клиенту нейтральное сообщение («Ошибка создания отчёта»). **Реализовано:** в ответах возвращается только нейтральное сообщение; код ошибки по-прежнему пишется в лог и в audit (payload для внутреннего разбора).

---

### 3. ~~Эндпоинт /_meta (низкий риск)~~ ✅ Исправлено

**Файл:** `index.js`, GET `/_meta`.

**Суть:** Возвращает хост/порт/базу (без пароля), флаги Firebase, `allowedOrigins`, версии приложения и т.д. Упрощает разведку окружения.

**Рекомендация:** в production либо отключить `/_meta`, либо ограничить доступ (например, по IP или по внутреннему заголовку) и не отдавать чувствительные флаги наружу. **Реализовано:** в production GET `/_meta` возвращает 404; в development эндпоинт по-прежнему доступен для диагностики.

---

### 4. SSL для подключения к БД: `rejectUnauthorized` (средний риск в контексте MITM) — частично учтено

**Файл:** `db.js`.

**Суть:** Для облачной БД использовалось жёстко заданное `rejectUnauthorized: false`, что отключает проверку сертификата сервера и допускает MITM при компрометации сети.

**Рекомендация:** по возможности использовать сертификат CA (например, от облачного провайдера) и включить проверку (`rejectUnauthorized: true`); если отключение необходимо — зафиксировать это в документации и в политике безопасности. **Сделано:** в `db.js` добавлен комментарий о риске и поддержка переменной окружения `PGSSL_REJECT_UNAUTHORIZED=true` для включения проверки сертификата при наличии настроенного CA.

---

### 5. CORS: запросы без заголовка Origin (по замыслу)

**Файл:** `index.js`, настройка CORS.

**Суть:** Запросы без `Origin` принимаются (для мобильных и десктопных клиентов). В production при этом всё равно требуется токен для доступа к данным.

**Рекомендация:** оставить как есть; при появлении чисто web-клиента можно ужесточить политику и не разрешать отсутствие `Origin` для браузеров.

---

### 6. Flutter (Web): токен в SharedPreferences (известный риск)

**Файл:** `lib/services/storage_service.dart`.

**Суть:** На платформе web токен хранится в SharedPreferences; при XSS в приложении токен может быть прочитан скриптом.

**Рекомендация:** в коде уже есть комментарий об этом. Для повышения безопасности web-версии рассмотреть выдачу и хранение токена в httpOnly cookie через бэкенд и не хранить JWT в JS-доступном хранилище.

---

### 7. Логирование в development (низкий риск)

**Файлы:** `middleware/auth.js`, ряд контроллеров.

**Суть:** В `NODE_ENV === 'development'` в логи попадают пути запросов, наличие заголовка Authorization, идентификаторы пользователей. Это приемлемо для разработки, но не должно попадать в production-логи.

**Рекомендация:** убедиться, что в production `NODE_ENV=production` и что в логах не выводятся токены, пароли и лишние PII (сейчас в production детали auth не логируются — хорошо).

---

### 8. Rate limit на /healthz (низкий риск)

**Файл:** `index.js`.

**Суть:** GET `/healthz?warm=1` выполняет запрос к БД. Глобальный rate limit (600 запросов / 15 мин) частично смягчает злоупотребление, но целенаправленная нагрузка может создавать лишние соединения к БД.

**Рекомендация:** при необходимости ограничить частоту вызовов `/healthz` с `warm=1` отдельным лимитером или кэшировать результат на несколько секунд.

## Проверенные сценарии (кратко)

| Область | Результат |
|--------|-----------|
| Доступ к чужим чатам/сообщениям | Проверка членства в чате (`ensureChatMember`) перед операциями с сообщениями |
| Доступ к чужим отчётам | Ограничение по `created_by`; суперпользователь — только через `requireSuperuser` |
| Доступ к студентам/урокам | Только свои или через `requirePrivateAccess`/`requireSuperuser`; привязка к `teacher_id`/`created_by` |
| Сброс пароля / админка | Только при `requireSuperuser` |
| SQL-инъекции | Параметризованные запросы; конкатенация пользовательского ввода в SQL не используется |
| XSS в сообщениях/никнеймах | Санитизация контента и отображаемых строк; SVG не принимается в изображениях |
| Загрузка файлов | Ограничение типов и размера; безопасные имена файлов |
| Секреты в коде/репозитории | Не обнаружены; скрипт проверки staged-файлов на секреты |
| Хранение токена (Flutter) | Secure storage на mobile/desktop; на web — замечание см. выше |
